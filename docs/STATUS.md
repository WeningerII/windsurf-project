## Current Product Snapshot

- 7 registered systems are live in the registry.
- Netlify is the canonical deployment target.
- The app is local-first with an optional cloud sync layer. Signed-out and unconfigured paths remain pure browser-local (IndexedDB primary, localStorage fallback, dual-write persistence). Signed-in users on a configured Supabase project get per-user document and campaign sync with offline queueing, realtime change propagation, and exponential-backoff retry. See `docs/rfc/001-backend-sync.md` for the shipped design.
- Browser-local scenes are now visible through a manual scene/grid manager. Users can create local scenes, associate them with campaigns/systems, place and move character/manual tokens through typed scene events, seed queued loader-backed D&D 5e monster encounters with party-level XP preview, manage terrain or hazard markers, set initiative, and import/export scene documents without Supabase or provider keys.
- The scene runs a deterministic, seeded auto-combat engine over the shared rules IR: an N-participant round driver resolves attacks, saves, area effects, conditions, concentration, death saves, and opportunity attacks RAW across all 7 systems, replayably from `(scene, seed)`.
- Combat geometry is fully three-dimensional on the flat top-down grid: flight movement, falling damage, and elevation-aware line-of-sight, cover, and area effects, with a multi-floor "view level" selector. Destructible `object` tokens block line of sight until felled.
- A deterministic, legible tactical AI drives the auto-round (utility scoring + a pluggable strategist seam, no ML in the hot path): threat/influence-map positioning — safe approach, flanking, cover-seeking, kiting — and expected-value focus-fire toward reliable kills.
- Combat narration has an optional server-gated LLM path (a Netlify function holding the key) that enriches an already-resolved round into prose and never authors mechanics; with no key the deterministic narrator stands in.
- Every system runs through a deterministic creation legality gate: a per-system validator judges a build for catalog-valid choices, range/cap/budget legality, and structural integrity, reporting structured issues without mutating the document (errors block; warnings flag incomplete-but-legal choices). All 7 systems now expose a validator through the registry's `validateDocument` seam.
- Prompt-driven character creation is live for all 7 systems. A prompt box parses a free-text description, a deterministic per-system creator drafts a complete build from the same catalogs/templates the sheets use, the engine derives stats, and the validator confirms legality — producing a finished, rules-legal sheet (or the system's equivalent derivation). The builds are substantively complete: 5e fills skills and a caster's starting cantrips/spells; PF2e applies ancestry/background/class boosts plus the four free level-1 boosts; D&D 3.5e/PF1e assign class-skill ranks within the per-skill cap; M&M lays a PL-scaled brick with traits, defenses, a Damage power, and combat advantages inside budget and every PL cap; Daggerheart fills the fixed trait array, starting domain cards, and experiences.
- The prompt box is LLM-authored across all 7 systems via a server-gated build gateway (a Netlify function holding the key, model id from operator env). The browser hands the model a compact, machine-readable options manifest (the real catalog names + the system's structural rules) and it returns a full BuildSpec, which each per-system creator re-validates against the real catalogs and uses where it resolves, falling back per field to its deterministic logic; the engine then derives and the validator judges. The authored surface is exhaustive across the data model, not a curated subset: class and full **multiclass** (per-class levels + subclasses), species/ancestry/heritage/race, background, ability allocation, **skill proficiencies/ranks/trainings**, spells (gated to the character's real top castable level, computed the same way the engine derives slots), feats (applied with their automation), and the M&M/Daggerheart equivalents (powers, defenses, advantages, domain cards, experiences). A validate-and-repair loop closes the fidelity gap with **universal no-silent-drops**: validator errors plus *every* author pick that didn't resolve — class/species/background names and also each spell/feat/skill/power/subclass/heritage/domain-card the creator skipped — are reported back to the model with its previous selections so it corrects its own attempt, for a bounded number of rounds, keeping the best result. So the model genuinely designs the character (a "Batman" becomes a stealthy Rogue, not a keyword-nearest Bard) — and is told to fix a hallucinated pick rather than have it quietly replaced — but it can never produce an illegal sheet, and with no key the prompt box degrades to the deterministic builder. This is the BEFORE-mechanics half of "AI proposes, the rules decide" (the combat narrator is the AFTER half). A local dev runner (`scripts/ai-create.ts`) exercises the live path with a key.
- Loader-backed counts, support levels, and source-filtered categories are generated from the runtime data/reporting path.
- Spell catalogs across 5e, D&D 3.5e, PF1e, and PF2e now share normalized index surfaces with alias-safe lookup, legacy d20 source-backed save/component/casting metadata coverage, and a cross-system identity regression matrix. The only D&D 3.5e spell source-blocked exclusions are `bleed-35e`, `mass-misdirection-35e`, and `reversal-of-fortune-35e`; PF1e source-backed rows without a Saving Throw line are explicit regression fixtures.
- Spell-catalog completeness is an explicit, now-visible track: the LLM author's no-silent-drops reporting surfaces any spell a player names that the catalog lacks, turning silent gaps into measurable signal, and that signal has been chased system by system. The 2014 SRD catalog gained 72 missing SRD spells (Fireball, Lightning Bolt, Cone of Cold, Moonbeam, …; 244 → 316), and 2024 gained its druid omissions (Druidcraft, Moonbeam, Spike Growth, Wind Wall; 320 → 324). The 3.5e arcane (Sorcerer/Wizard) list — previously almost entirely absent (no level-1 arcane at all, 2 level-2 spells) — was filled to the SRD across all nine spell levels, and the Bard list (49 spells, no level-1) was completed for levels 0–6 (428 → 574, ~350 spells added or class-merged). PF1e was then expanded to a full catalog: as a 3.5e derivative it shares almost its entire list, so the complete Sorcerer/Wizard, Cleric, Druid, and Bard spell lists were ported under CRB conventions (134 → 583, all classes comprehensively covered). PF2e gained the core CRB spells it was missing across cantrips through rank 9 (Harm, Bane, Sleep, Charm, Stabilize, Divine Lance, Detect Magic, Mage Armor, Shatter, Phantasmal Killer, Weird, …; 143 → 170). Every caster class across 5e (both editions), 3.5e, and PF1e now has a balanced per-level distribution, and an iconic-coverage sweep confirms every must-have spell resolves for its real class in all five casting systems; PF2e covers cantrips/common play with the iconic higher ranks. All additions are SRD/CRB-accurate with no duplicate names/ids and no stub descriptions. (Subclass/heritage breadth remains SRD-scoped — one subclass per 5e class per SRD 5.1 — and is governed by the open-content policy, not this spell track.)
- Shared controller/section-host convergence is shipped across 5e, PF2e, legacy d20, M&M, and Daggerheart. Future work here is maintenance against existing host/controller contracts, not another decomposition push.
